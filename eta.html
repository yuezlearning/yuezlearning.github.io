ETA: Estimated Time to Achievement

A Python-based web scraping tool that estimates total study time for WGU Academy courses by analyzing every page — reading time, video durations, interactive activities, quizzes, and embedded resources.

Role

Developer

Duration

2026 — ongoing

Stack

Python, Playwright, yt-dlp

Code

GitHub ↗

01Overview

WGU Academy courses vary widely in length and complexity, but students have no reliable way to estimate how long a course will actually take. ETA solves this by crawling every page of a course and computing time estimates based on content type.

The tool runs headlessly using Playwright, authenticates against WGU Studio, and produces structured JSON summaries that teams can use for scheduling, advising, and content planning.

02Problem

Course teams and academic advisors needed accurate time estimates to:

01

Set student expectations — learners need to know upfront how many hours a module requires

02

Balance course workload — identify modules that are disproportionately heavy or light

03

Support scheduling — enable advisors to recommend realistic pacing plans

Manual estimation was slow, inconsistent, and didn't account for embedded videos, interactive blocks, or external resources.

03Estimation

Each content type has a tailored estimation strategy:

Content Type Estimation Rule
Text Word count ÷ 175 wpm
Video (embedded) Actual duration via YouTube/Vimeo lookup
Video (external) Duration from text, or 10 min default
Interactive blocks 3 minutes each
Problem/quiz 0.75 min per question
Case study / Simulator 8 minutes
Reading (banner) 1.5 min × page count
Podcast Duration from text, or 10 min default

04Architecture

The tool runs a 3-stage pipeline with parallel browser instances:

Stage 1: Page Scraping

Visits every vertical in parallel (5 workers), extracts word counts, videos, interactives, banners, and problem blocks.

Stage 2: YouTube Lookup

Resolves durations for embedded YouTube videos via yt-dlp.

Stage 3: Vimeo Lookup

Resolves durations for embedded Vimeo videos via postMessage handshake.

Concurrency Model

5 parallel Playwright browser instances scrape pages concurrently, then video lookups run sequentially to avoid rate-limiting.

05Output

The tool produces structured JSON for each course:

timer_summary.json

Course title, total time, and per-page times — the quick reference.

timer_details.json

Full breakdown including word counts, video sources, banner types, etc.

Batch Mode

List courses in course.txt and run scrape_all.py — it iterates through every course, runs the full pipeline, and saves per-course JSON files.

06Reflection

Automation as a learning design tool

What started as a dev utility became a learning design asset. The time data surfaced content balance issues that human review had missed — modules with 3× the expected workload, videos linked but never watched, and interactive blocks that added time without adding value.

Dealing with fragile scraping targets

Open edX pages are heavily JavaScript-rendered with inconsistent DOM structures across courses. Building resilient selectors and graceful fallbacks taught me to design for uncertainty — a mindset that applies equally to UX and engineering.

Next steps

Exploring a dashboard layer that visualizes time distribution across modules, flags outliers, and integrates with WGU's content planning workflow. Also considering LRPS (Learner-Reported Pacing Scores) to validate estimated vs. actual time.
