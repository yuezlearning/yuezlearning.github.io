ETA: Estimated Time to Achievement

A Python-based web scraping tool that estimates study time for WGU Academy courses by analyzing every page — reading time, video durations, interactive activities, quizzes, and embedded resources.

Role

Instructional Technologist

Duration

05/2026

Stack

Python, Playwright, Kiro, Github

01Overview

WGU Academy courses vary widely in length and complexity, but students and advisors have no reliable way to estimate how long a course will actually take. ETA solves this by programmatically scraping every page of a course and computing granular time estimates based on content type — text, video, interactive activities, quizzes, and embedded resources.

The tool runs headlessly using Playwright, authenticates against WGU Studio, and produces structured JSON summaries that course teams can use for scheduling, student advising, workload balancing, and content planning. What previously required hours of manual estimation now runs in under 5 minutes.

03Problem

Course teams and academic advisors needed accurate, data-driven time estimates to:

01

Set student expectations — learners need to know upfront how many hours a module requires so they can plan their study schedules effectively

02

Balance course workload — identify modules that are disproportionately heavy or light, enabling designers to redistribute content for a smoother learner experience

03

Support scheduling — enable advisors to recommend realistic completion pacing guides based on actual content volume rather than guesswork

Manual estimation was slow, inconsistent, and couldn't account for embedded videos, interactive blocks, or external resources. Different team members would arrive at different numbers for the same course — making planning unreliable.

04Estimation

I collaborated with our Learning Experience Designers team to define estimation rules grounded in pedagogical research and real student behavior. Each content type has a tailored strategy:

Content Type Estimation Rule
Text Word count ÷ 175 wpm
Video (embedded) Actual duration via YouTube/Vimeo lookup
Video (external) Duration from text, or 10 min default
Interactive blocks 3 minutes each
Problem/quiz 0.75 min per question
Case study / Simulator 8 minutes
Reading (banner) 1.5 min × page count
Podcast Duration from text, or 10 min default

05Architecture

The tool runs a 3-stage pipeline with parallel browser instances, designed for both speed and reliability:

Stage 1: Page Scraping

Visits every vertical in parallel (5 workers), extracts word counts, video embeds, interactive blocks, banners, and problem sets. Handles dynamic content loading and JavaScript-rendered elements.

Stage 2: YouTube Lookup

Resolves actual durations for embedded YouTube videos via yt-dlp, replacing placeholder estimates with precise runtime data.

Stage 3: Vimeo Lookup

Resolves durations for embedded Vimeo videos via postMessage handshake, handling private and unlisted videos that don't expose metadata through standard APIs.

Concurrency Model

By introducing a concurrency workflow with 5 parallel Playwright browser instances, the time to estimate one full course decreased from over 10 minutes to less than 5 minutes — making it practical to run as part of daily QA routines.

06Output

The tool produces structured JSON for each course:

timer_summary.json

Course title, total time, and per-page times — the quick reference.

timer_details.json

Full breakdown including word counts, video sources, banner types, etc.

Batch Mode

List courses in course.txt and run scrape_all.py — it iterates through every course, runs the full pipeline, and saves per-course JSON files.

AI automating everyday workflows

This project reinforced a broader lesson: using AI to automate repetitive workflows creates compounding returns. The time saved on estimation is now redirected toward higher-value work — improving content quality, refining learning experiences, and supporting students more effectively.
